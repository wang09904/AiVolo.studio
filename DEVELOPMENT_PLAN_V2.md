## 1. 文档用途

本项目正式开发实施阶段由 AI 全流程执行。AI 不只是写代码，还必须承担专业软件工程中的任务拆分、并行调度、串行依赖管理、专业 review、测试验证、质量门禁和交付把关。

本文件用于约束：

- 每个阶段做什么。
- 每个任务当前状态如何。
- 哪些任务可以并行。
- 哪些任务必须串行。
- 哪类专业任务交给哪个 Agent。
- 每个阶段和模块如何验收。
- 提交 Git 和部署前必须经过哪些检查。

## 4. AI 开发编排模式

### 4.1 主执行 Agent 职责

主执行 Agent 负责整体交付，不得只做局部编码。

职责包括：

- 阅读并遵守 `SPEC_V2.md`。
- 拆分任务模块。
- 判断并行与串行关系。
- 指派专业 Agent。
- 管理 Agent 输出边界。
- 整合代码和文档。
- 处理冲突。
- 执行质量门禁。
- 最终验收和交付说明。

### 4.2 编排原则

1. **专业任务交给专业 Agent**
   - UI、前端、后端、数据库、支付、SEO、测试、代码审查、安全审查分别由专业 Agent 负责。

2. **能并行的任务并行**
   - 页面、后端 API、数据库、法律页面、测试方案、SEO 内容等可在边界清晰时并行。

3. **必须串行的任务严格串行**
   - 数据库 schema 先于 API。
   - API 合约先于前端完整接入。
   - 计费规则先于支付和退款实现。
   - UI 方向先于大规模视觉精修。

4. **主执行 Agent 统一验收**
   - 专业 Agent 的产出不能直接视为完成，必须由主执行 Agent 汇总、联调、验证。

5. **文件边界明确**
   - 并行任务必须提前定义负责文件和模块，避免多个 Agent 修改同一核心文件造成冲突。

## 6. 工程质量门禁

开发任务不得以“代码写完”为完成标准。每个模块必须经过实现、联调、自测、专业 review、修复、回归和最终验收。

### 6.1 标准交付流程

每个功能模块必须按以下流程执行：

1. **需求确认**
   - 对照 `SPEC_V2.md`。
   - 明确功能范围。
   - 明确不做事项。
   - 明确验收标准。

2. **技术设计**
   - 明确页面、接口、数据库、外部服务。
   - 明确失败路径。
   - 明确并行/串行依赖。
   - 高风险模块先写实现方案再编码。

3. **实现开发**
   - 分配给专业 Agent。
   - 控制文件边界。
   - 不修改无关文件。
   - 不扩大任务范围。

4. **自测与联调**
   - 功能开发完成后必须自测。
   - 前后端相关功能必须联调。
   - 认证、积分、支付、存储必须跑完整闭环。

5. **专业 Review**
   - 前端/UI 由 frontend/design Agent review。
   - 后端/API/数据库由 backend/database Agent review。
   - 支付、积分、认证、数据一致性由 security/critic Agent review。
   - 代码质量由 code-reviewer Agent review。
   - 用户流程由 QA Agent review。

6. **修复与回归**
   - Review 发现的问题必须修复。
   - 修复后必须重新验证。
   - 不允许带已知 P0/P1 问题进入提交。

7. **提交前检查**
   - 查看 git diff。
   - 确认无密钥、环境变量、测试垃圾文件。
   - 运行 build、lint、typecheck 和必要测试。
   - 确认改动范围与任务一致。

8. **Git 提交**
   - 只提交相关文件。
   - commit message 清晰说明改动。
   - 提交前必须完成最终 review。

9. **部署后验证**
   - 检查线上关键路径。
   - 登录、生成、积分、下载、核心页面必须可用。
   - 出现问题必须记录、修复或回滚。

### 6.2 质量门禁表

| 门禁 | 触发时机 | 负责 Agent | 必须通过标准 |
| --- | --- | --- | --- |
| 需求门禁 | 开发前 | product/planner | 范围、验收、不做事项明确 |
| 架构门禁 | 高风险模块前 | architect/critic | 数据流、依赖、失败路径清楚 |
| 代码门禁 | 功能开发后 | code-reviewer | 无明显 bug、无无关改动 |
| 安全门禁 | auth/payment/credits/storage | security/critic | 无越权、无密钥泄露、账务安全 |
| 数据门禁 | DB/RPC/RLS | database/backend | schema、RLS、原子操作正确 |
| UI 门禁 | 页面开发后 | designer/frontend | 符合视觉方向，基础响应式可用 |
| QA 门禁 | 阶段完成前 | qa/test-engineer | 核心用户路径跑通 |
| 提交门禁 | git commit 前 | git/code-reviewer | diff 干净，验证通过 |
| 部署门禁 | 上线后 | qa/devops | 线上关键流程可用 |

### 6.3 禁止事项

- 禁止未对照 SPEC 直接开发。
- 禁止功能未自测就标记完成。
- 禁止未 review 就提交 Git。
- 禁止提交无关文件、临时文件、密钥、环境变量。
- 禁止在支付、积分、认证、数据库模块中跳过安全审查。
- 禁止因为时间紧跳过 build、lint、typecheck。
- 禁止多个 Agent 同时修改同一核心文件而不协调。

### 8.2 批次 2A：支付与账务闭环

目标：

- Creem.io 真实支付。
- 订阅状态。
- webhook 防重复。
- 积分发放。
- 积分流水记录 `credit_unit_value_usd`。
- 退款按已消耗有价积分折算金额。
- 退款申请入口放在 `/refund` 和 `/account/billing`，不放在生成历史或生成结果页。
- 订阅升级按当前实付价补差额，不重复收取完整新套餐费用。
- 升级后新增积分必须作为 `subscription_upgrade_grant` 独立批次记录，不改写旧套餐积分批次。
- 订阅有效期内禁止主动降级；用户只能等当前订阅到期恢复免费后重新购买低级订阅。
- 退款后积分扣回或权益回收。

质量门禁：

- 必须经过 security/critic review。
- 必须经过 database/backend review。
- 必须有 webhook 重放和重复事件处理策略。
- 必须验证免费积分价值为 0、订阅积分按套餐单积分价值折算退款。
- 必须验证用户日常界面不显示当前可退余额或剩余可退款金额，但后台保留该字段或可实时计算。
- 必须验证用户进入退款申请/审核流程后，可以展示可退款金额和计算依据。
- 必须验证退款入口低强调展示在退款政策页和账单/订阅管理页。
- 必须验证 Lite 到 Pro、月付到年付等升级场景的补差价计算。
- 必须验证升级后旧套餐积分批次和升级补差积分批次分开记录，并按各自 `credit_unit_value_usd` 参与退款审计。
- 必须验证当前订阅有效期内没有降级入口，后端也拒绝降级请求。

### 8.3 批次 2B：模型与视频能力

- 支持约 10 个模型。
- 每个模型必须配置用户侧积分消耗参数。
- 支持图生图、文生视频、图生视频。
- 视频任务使用异步轮询。
- 高可用路由：主动健康检查 + 被动失败切换。

质量门禁：

- 必须验证失败切换。
- 必须验证积分失败退还。
- 必须验证不同模型按配置扣除不同积分。
- 必须验证视频任务 pending/running/completed/failed 状态。

### 8.4 批次 2C：多语言与 SEO/GEO

## 11. 数据库规划

### 11.1 第一阶段核心表

| 表 | 说明 |
| --- | --- |
| users | 用户资料和积分余额 |
| credits_transactions | 积分流水，必须支持退款审计 |
| generations | 生成任务、结果、积分消耗和金额折算 |
| subscriptions | 订阅占位和后续扩展 |

### 11.2 第二阶段扩展表

| 表 | 说明 |
| --- | --- |
| models | 模型配置 |
| templates | 模板配置 |
| webhooks_log | Creem webhook 防重复 |
| feedback | Bug 和建议反馈 |
| generation_assets | 生成结果资产管理 |
| generation_credit_usages | 单次生成任务消耗多批积分时的分摊明细 |

`models` 表或等价模型配置必须包含用户侧积分消耗参数，例如 `generation_type`、`quality_tier`、`credits_per_image`、`credits_per_second`、`min_credits_per_task`、`is_active`。第一阶段单模型文生图按 1 积分/张扣除；第二阶段多模型按配置扣除。

`credits_transactions` 表必须支持退款审计所需字段，例如 `credit_source`、`plan_id`、`from_plan_id`、`upgrade_order_id`、`credit_unit_value_usd`、`billing_period_start`、`billing_period_end`、`generation_id`。退款金额应按已消耗有价积分折算，而不是另按图片张数或视频秒数建立独立美元单价。

订阅升级产生的新增积分必须记录为 `credit_source = subscription_upgrade_grant` 的独立积分批次。旧套餐未使用积分保留原 `plan_id` 和原 `credit_unit_value_usd`，不得被升级操作改写。

`generations` 表必须支持生成历史和退款审计所需字段，例如 `generation_type`、`model_id`、`duration_seconds`、`status`、`credits_charged`、`credit_unit_value_usd`、`used_value_usd`、`credit_transaction_id`、`asset_url`、`thumbnail_url`、`completed_at`。第二阶段支付和退款上线前必须补齐这些字段。

如果一次生成任务可能消耗多批不同来源或不同单价的积分，必须用 `generation_credit_usages` 或等价结构记录分摊明细，确保退款金额可追溯。

| --- | --- |

| cache_hits | 缓存命中记录 |

## 14. 风险与外部依赖

| 风险 | 阶段 | 处理方式 |
| --- | --- | --- |
| 用户真实需求未验证 | 第二阶段前 | 等待用户调研结果，避免垂类写死 |
