## 1. 产品定位

**AiVolo.studio** 是面向海外市场的 AI 图片/视频娱乐创作产品。

它不是严肃的企业级科技 SaaS，也不是面向专业影视团队的复杂生产工具。它的核心方向是：让普通用户、短视频创作者、电商经营者用更低门槛生成可用于社交传播、营销展示和内容创作的图片与视频。

### 1.1 核心价值

- **低门槛**：用户不需要理解复杂模型参数，也不需要掌握专业 prompt 技巧。
- **娱乐感**：产品要像消费级娱乐创作工具，而不是冷冰冰的 AI 工具站。
- **高效率**：用户输入简单需求后，快速获得可下载、可复用的图片或视频结果。
- **聚合能力**：第二阶段开始接入多个模型和 API 聚合供应商，降低用户在多个 AI 产品之间切换和订阅的成本。
- **SEO/GEO 增长能力**：通过模型详情页、模板页、多语言内容和垂类页面承接搜索与生成式搜索流量。

### 1.2 目标市场

优先面向海外市场：

1. 欧洲：西班牙、葡萄牙、法国、意大利、德国等。
2. 南美：巴西、墨西哥等。
3. 东南亚：印尼、越南、泰国等。
4. 北美：美国、加拿大等。

### 1.3 目标用户

- 短视频创作者：TikTok、YouTube Shorts、Instagram Reels 等平台创作者。
- 普通消费者：被 viral 内容吸引，希望低成本尝试 AI 图片/视频生成。
- 小型电商卖家：需要商品图、营销图、短视频素材。
- 室内设计、展示类内容用户：第二阶段模板场景之一。

### 1.4 当前待验证假设

以下事项不阻塞第一阶段 MVP，但会影响第二阶段和后续增长策略：

- 当前规划的功能是否对应真实用户高频需求。
- 图片/视频生成应优先切入哪些垂类场景。
- 模板、模型详情页、SEO/GEO 内容应围绕哪些具体需求组织。

用户后续会按课程调研方法补充真实用户需求验证结果。文档后续应根据该结果调整垂类方向和功能优先级。

## 2. 项目节奏

### 2.1 时间口径

本项目有两个不同层级的时间口径，后续不得混淆：

- **第一阶段 MVP**：目标是在 **7 天内完成上线部署**。
- **第二阶段主体功能**：目标是在 **7-10 天内完成主体产品能力**。
- **第三阶段**：学习小组进入 SEO/GEO 推广、内容投放和运营验证，同时补齐少量第三阶段功能。SEO/GEO 相关的网站页面开发应在第二阶段完成。
- **21 天周期**：指整个学习陪跑活动的总周期，包含开发、部署、运营营销和初步结果验证，不是第一阶段开发周期。

### 2.2 阶段定义

| 阶段        | 目标         | 主要内容                          |
| --------- | ---------- | ----------------------------- |
| 第一阶段 MVP  | 7 天内上线     | 登录、文生图、积分、订阅 UI、三件套、账户页、部署    |
| 第二阶段主体功能  | 7-10 天     | 多模型、图生图、视频生成、模板、多语言、真实支付、模型广场、模板 SEO 页面 |
| 第三阶段推广与补齐 | 推广运营阶段     | 电商模板、智能缓存层、SEO/GEO 运营推广配合       |
| 第四阶段可选    | 后续增强       | 数据分析                          |

### 4.4 模型详情页结构

每个模型应有独立 SEO 页面：

1. 第一屏：SEO 内容，说明模型能力、适用场景、示例。
2. 第二屏：内嵌生成框，预设当前模型。
3. 后续内容：示例、FAQ、相关模型、相关模板。

### 4.5 模板系统与 SEO 页面结构

模板系统同样是 SEO/GEO 的核心承接面，不应只作为站内功能入口。

模板系统没有三类模板之上的总列表页。首页左侧边栏直接提供三类模板入口，每一类都是一级入口：

```text
首页左侧边栏
└── 模板区
    ├── 热门模板 → /templates/viral
    ├── 室内设计模板 → /templates/interior
    └── 电商模板 → /templates/ecommerce

模板分类列表页
└── 展示该分类下所有模板
    └── 点击某个模板 → /template/[slug]
        └── 模板详情页
            └── 点击 Use this template → 进入生成任务
```

模板分类列表页应包含：

1. 分类 SEO 标题与说明。
2. 该分类适用人群和使用场景。
3. 模板卡片列表。
4. 热门模板推荐。
5. FAQ。

模板详情页应包含：

1. 模板名称和面向的具体生成需求。
2. SEO 说明内容。
3. 示例图/视频。
4. 参数表单或预设 prompt。
5. 使用该模板生成按钮。
6. 相关模板和相关模型。
7. FAQ。

第一阶段只实现文生图：

- 选择模型，第一阶段只有一个模型但仍保留下拉框。
- 选择图片比例。
- 后台默认质量为 1080P，不在第一阶段前台暴露质量参数。
- 调用 OpenAI 兼容 API。
- 返回图片结果。
- 存储到 Cloudflare R2。
- 返回可展示和可下载的签名 URL。

### 6.2 图生图与图生视频

第二阶段实现：

- 用户上传参考图。
- 参考图进入产品封装层处理。
- 产品后端再调用 API 供应商。
- 参考图不直接上传给供应商作为用户侧直连流程。
- 第二阶段只做单图生视频。
- 不做多图生视频和首尾帧生视频。

### 6.3 参考图策略

参考图用于一次生成任务，**用完即删，不长期保存**。

建议限制：

- 最大 5MB。
- 支持 JPG、PNG、WebP。
- 可在前端做压缩提示或压缩处理。

### 6.6 生成历史与审计记录

由于退款金额按“已消耗有价积分”折算，生成历史必须具备审计能力，不能只记录 prompt、缩略图和下载链接。

生成历史需要同时服务三个目标：

1. 用户侧查看：让用户看到自己生成过什么、何时生成、使用了哪个模型、消耗了多少积分、结果是否可下载。
2. 运营侧分析：后续用于分析模型使用、成本、转化和用户偏好。
3. 退款侧审计：能够追溯某次生成任务消耗了哪一批积分、这些积分的单积分金额价值是多少。

用户侧不需要看到所有审计字段。生成历史需要分成“用户可见信息”和“后台审计信息”两层。

用户在账户页/生成历史中可见：

| 字段 | 说明 |
| --- | --- |
| 生成时间 | 任务创建或完成时间 |
| 生成类型 | 文生图、图生图、文生视频、图生视频 |
| 模型名称 | 用户实际使用的模型名称 |
| Prompt 摘要 | 用户输入内容的摘要或完整 prompt |
| 缩略图/结果 | 生成结果预览和下载入口 |
| 任务状态 | 生成中、成功、失败 |
| 消耗积分 | 本次任务扣除多少积分 |
| 折算消耗金额 | 本次任务消耗积分对应的金额价值 |

后台和运营人员可见：

| 字段 | 用途 |
| --- | --- |
| 积分批次 | 判断积分来源和单价 |
| 积分流水 ID | 审计积分发放、消耗、退回 |
| 单积分金额价值 | 退款金额折算 |
| used_value_usd | 已使用金额 |
| refundable_balance_usd | 当前可退款余额，默认不在用户日常界面展示 |
| 供应商和模型 ID | 成本分析和问题排查 |
| final_prompt | 缓存、质量分析和模型调试 |
| 错误信息 | 失败排查 |
| 账期归属 | 订阅和退款审计 |

因此，前端用户历史记录页面应保持简洁；复杂字段只进入后台管理、运营分析和退款审计，不应全部暴露给用户。

`generations` 表应记录：

| 字段 | 说明 |
| --- | --- |
| `id` | 生成任务 ID |
| `user_id` | 用户 ID |
| `generation_type` | text-to-image / image-to-image / text-to-video / image-to-video |
| `model_id` | 实际使用模型 |
| `prompt` | 用户输入 prompt |
| `final_prompt` | 后端处理后的 prompt，第二阶段可用 |
| `aspect_ratio` | 图片/视频比例 |
| `duration_seconds` | 视频时长，图片为空 |
| `status` | pending / running / completed / failed |
| `credits_charged` | 本次任务实际扣除积分 |
| `credit_unit_value_usd` | 本次消耗积分对应的单积分金额价值 |
| `used_value_usd` | `credits_charged × credit_unit_value_usd` |
| `credit_transaction_id` | 对应积分流水 |
| `asset_url` | 生成结果地址 |
| `thumbnail_url` | 缩略图地址 |
| `error_message` | 失败原因 |
| `created_at` | 创建时间 |
| `completed_at` | 完成时间 |

`credits_transactions` 表应记录每一次积分发放、消耗、退回和退款调整。生成任务扣积分时，`generations.credit_transaction_id` 必须能关联到对应的积分流水。

如果一次生成任务消耗了多批不同来源或不同单价的积分，应记录可审计的分摊明细。实现上可以使用 `generation_credit_usages` 表：

| 字段 | 说明 |
| --- | --- |
| `generation_id` | 生成任务 |
| `credit_transaction_id` | 被消耗的积分批次或流水 |
| `credits_used` | 从该批次消耗多少积分 |
| `credit_unit_value_usd` | 该批次单积分金额价值 |
| `used_value_usd` | 该批次折算金额 |

第一阶段可以先记录单模型文生图的 `credits_charged` 和基础历史字段；第二阶段支付和退款上线前，必须补齐金额价值、积分来源和分摊审计字段。

## 7. 积分、订阅与支付

### 7.1 免费积分

- 新注册用户赠送 20 积分。
- 积分用于体验第一阶段文生图能力。

### 7.2 积分扣除

生成任务必须通过原子流程处理：

1. 检查用户余额。
2. 预扣积分。
3. 创建 pending 生成任务。
4. 调用模型生成。
5. 成功后标记 completed 并写入结果。
6. 失败后标记 failed 并退还预扣积分。

### 7.3 用户侧积分消耗规则

积分是产品内部统一计量单位，既用于用户生成图片/视频时扣除 credits，也用于订阅退款时折算已使用金额。

因此，用户侧积分消耗和后端退款金额计算应保持同一套数据口径：

- 用户生成任务时，按模型配置扣除积分。
- 退款时，不再按“生成图片张数/视频秒数”另起一套固定美元单价，而是按用户实际消耗的有价积分折算已使用金额。
- 免费赠送积分、补偿积分、订阅付费积分必须在积分流水中区分来源，因为它们的退款价值不同。

由于第二阶段具体接入模型尚未最终确定，积分消耗必须设计为可配置，而不是写死在代码里。每个模型应在 `models` 配置中维护消耗参数。

建议模型定价配置字段：

| 字段 | 说明 |
| --- | --- |
| `model_id` | 模型 ID |
| `generation_type` | text-to-image / image-to-image / text-to-video / image-to-video |
| `quality_tier` | basic / advanced / premium |
| `credits_per_image` | 每张图片消耗积分 |
| `credits_per_second` | 每秒视频消耗积分 |
| `min_credits_per_task` | 单次任务最低消耗 |
| `is_active` | 是否启用 |

阶段默认积分消耗表：

| 操作类型 | 默认消耗积分 | 说明 |
| --- | --- | --- |
| 文生图（基础模型） | 1 积分/张 | 第一阶段默认规则；新用户 20 积分可体验约 20 张图 |
| 文生图（高级模型） | 3 积分/张 | 第二阶段接入高级图片模型后启用 |
| 图生图（基础模型） | 2 积分/张 | 第二阶段启用 |
| 图生图（高级模型） | 5 积分/张 | 第二阶段接入高级图生图模型后启用 |
| 文生视频（标准） | 10 积分/秒 | 第二阶段启用，具体按供应商模型成本可调整 |
| 图生视频（标准） | 15 积分/秒 | 第二阶段启用，具体按供应商模型成本可调整 |

扣分公式：

```text
图片任务消耗积分 = 输出图片数量 × 对应模型 credits_per_image
视频任务消耗积分 = 视频秒数 × 对应模型 credits_per_second
最终消耗积分 = max(计算结果, min_credits_per_task)
```

第一阶段只有单模型文生图，默认按 **1 积分/张** 扣除。第二阶段接入多模型后，前端应在用户提交生成前展示预计消耗积分，后端以模型配置为准执行原子预扣。

### 7.4 积分金额价值

付费订阅中的积分应有可追溯的金额价值：

```text
单积分金额价值 = 套餐有效月价格 / 每月发放积分
```

按当前定价与权益计算：

| 套餐 | 有效月价格 | 每月积分 | 单积分金额价值 |
| --- | --- | --- | --- |
| 免费版 | $0 | 20 | $0 |
| Lite 月付 | $15 | 300 | $0.0500 / 积分 |
| Lite 年付 | $10 | 300 | $0.0333 / 积分 |
| Pro 月付 | $29 | 800 | $0.03625 / 积分 |
| Pro 年付 | $14.5 | 800 | $0.018125 / 积分 |

年付产品虽然在支付页按全年金额扣费，但积分金额价值仍按月均价格和每月积分计算。例如 Pro 年付的有效月价为 $14.5，每月 800 积分，因此单积分价值为 $14.5 / 800 = $0.018125。

积分流水必须记录：

| 字段 | 说明 |
| --- | --- |
| `credits_delta` | 积分增减数量 |
| `credit_source` | free_signup / subscription_grant / refund_adjustment / admin_adjustment |
| `plan_id` | 产生积分的套餐 |
| `credit_unit_value_usd` | 当前这批积分的单积分金额价值 |
| `billing_period_start` | 归属账期开始 |
| `billing_period_end` | 归属账期结束 |
| `generation_id` | 消耗积分对应的生成任务 |

免费积分和补偿积分的 `credit_unit_value_usd` 为 0。订阅发放的积分按套餐记录对应的单积分金额价值。

### 7.11 订阅升级补差价规则

用户从当前订阅 A 升级到更高级订阅 B 时，应采用对用户友好的补差价规则：用户只需要按照当前价格和当前实际已支付金额补齐差额。

基础规则：

```text
升级需支付金额 = 目标套餐当前应付金额 - 当前套餐当前账期已支付金额
```

如果结果小于或等于 0，则不额外收费，但仍可更新套餐权益；是否立即发放差额积分由产品策略决定。

计算需要考虑：

| 字段 | 说明 |
| --- | --- |
| `current_plan_id` | 当前套餐 |
| `target_plan_id` | 目标套餐 |
| `billing_period_start` | 当前账期开始 |
| `billing_period_end` | 当前账期结束 |
| `amount_paid_current_period` | 当前账期实际已支付金额 |
| `target_plan_price_current` | 目标套餐当前应付金额 |
| `upgrade_amount_due` | 本次升级需补差价 |

示例：

```text
用户当前为 Lite 月付：已支付 $15
目标升级为 Pro 月付：当前价格 $29
升级需支付金额 = $29 - $15 = $14
```

年付升级也按当前有效价格计算。例如用户从 Lite 年付升级到 Pro 年付，应按当前年付价格或当前账期等效价格计算差额。具体由 Creem.io 支持的订阅升级能力决定；如果支付平台不支持原生 proration，则由后端生成一次性补差价订单。

升级成功后：

1. 更新订阅套餐。
2. 发放或调整目标套餐权益。
3. 记录升级订单。
4. 记录新增积分批次及对应 `credit_unit_value_usd`。
5. 保留原套餐已消耗积分的历史金额价值，不回写篡改旧流水。

### 7.12 升级后的积分来源界定

用户从 A 订阅升级到更高级的 B 订阅时，积分来源必须通过积分批次区分，不允许把 A 套餐积分和 B 套餐权益混成一笔。

核心规则：

1. A 订阅已发放但未使用的积分，仍然保留为 A 套餐积分批次。
2. 用户支付升级差额后，系统只发放 B 相比 A 增加的差额积分。
3. 新增积分记录为升级补差积分批次。
4. 退款时按每一批积分自己的 `credit_unit_value_usd` 分别折算。

示例：

```text
Lite 月付：$15 / 300 积分，单积分价值 $0.05
Pro 月付：$29 / 800 积分

用户从 Lite 月付升级到 Pro 月付：
升级补差金额 = $29 - $15 = $14
新增积分 = 800 - 300 = 500
升级补差积分单价 = $14 / 500 = $0.028
```

升级后当前账期积分批次：

| 积分批次 | 数量 | 来源 | 单积分价值 |
| --- | ---: | --- | ---: |
| Lite 原始积分 | 300 | Lite 月付订阅 | $0.05 |
| Pro 升级补差积分 | 500 | Lite -> Pro 升级补差 | $0.028 |

金额校验：

```text
300 × $0.05 + 500 × $0.028 = $15 + $14 = $29
```

升级补差积分批次应记录：

| 字段 | 说明 |
| --- | --- |
| `credit_source` | `subscription_upgrade_grant` |
| `from_plan_id` | 升级前套餐 |
| `plan_id` | 升级后套餐 |
| `upgrade_order_id` | 升级补差订单 |
| `credits_delta` | 目标套餐积分 - 原套餐积分 |
| `credit_unit_value_usd` | 升级补差金额 / 新增积分 |
| `billing_period_start` | 当前账期开始 |
| `billing_period_end` | 当前账期结束 |

积分消费顺序：

1. 当前账期付费积分优先。
2. 同一到期日下，优先消耗旧套餐批次，再消耗升级补差批次。
3. 免费积分、赠送积分、补偿积分最后消耗，且退款价值为 0。

这样可以避免用户使用了付费权益后，由于系统优先扣免费积分而仍获得完整退款。

禁止事项：

- 不允许让用户在同一账期重复支付完整新套餐价格。
- 不允许用户在订阅有效期内主动降级订阅。
- 不允许删除旧订单或旧积分流水。
- 不允许用新套餐积分单价重算历史已消耗积分。

### 7.13 订阅降级规则

订阅有效期内不允许用户主动降级订阅。

如果用户希望使用更低级别的订阅，必须：

1. 等待当前订阅自然到期。
2. 当前订阅失效后，账户恢复为免费用户。
3. 用户再重新选择并购买新的低级别订阅。

产品和后端都不提供“当前订阅期内从高级套餐降到低级套餐”的入口和接口。

原因：

- 避免当前账期权益、积分价值和退款计算复杂化。
- 避免用户先购买高级套餐获取权益，再中途降级套利。
- 保持订阅、积分批次、退款金额计算规则简单、清晰、可审计。

### 7.14 退款金额计价规则

退款金额计价规则必须基于积分流水，而不是另建一套与前端积分消耗割裂的图片/视频美元单价。

退款金额按已使用部分扣除后退还：

```text
退款金额 = 订单金额 - 已使用金额
已使用金额 = Σ(已消耗积分数量 × 该批积分的 credit_unit_value_usd)
```

退款计算不追踪单次 API 实际 token 或供应商成本，也不按图片张数/视频秒数另设退款单价。退款只关心用户消耗了多少积分，以及这些积分来自哪一个套餐、每个积分对应多少美元价值。

退款流程：

1. 用户申请退款。
2. 查询订单和订阅状态。
3. 查询该订单/账期内发放的付费积分批次。
4. 查询这些积分批次中已经被生成任务消耗的数量。
5. 按 `已消耗积分数量 × credit_unit_value_usd` 计算已使用金额。
6. 从订单金额中扣除已使用金额。
7. 剩余金额通过 Creem.io 原路退回。
8. 同步取消或更新订阅状态。
9. 扣除或失效未使用权益和积分。

示例：

```text
Pro 年付有效月价 = $14.5
每月积分 = 800
单积分价值 = $14.5 / 800 = $0.018125

如果用户在当前账期消耗 200 个 Pro 年付积分：
已使用金额 = 200 × $0.018125 = $3.625
```

这种计算方式的优势：

- 用户侧看到的是积分消耗。
- 后端退款也是按积分消耗折算。
- 免费积分价值为 0，不会错误扣减退款。
- 不同套餐的积分金额价值不同，能够反映用户实际购买价格。
- 用户较真时，规则可解释、可审计、可追溯。

### 7.15 退款申请入口

退款入口应当合规可找到，但不应在用户正常生成路径中过度突出，避免主动引导用户思考退款。

退款申请入口放置规则：

| 位置 | 说明 |
| --- | --- |
| `/refund` 退款政策页 | 页面底部提供申请方式和联系邮箱，第一阶段即可存在 |
| `/account/billing` 订阅管理页 | 第二阶段真实支付上线后，在付款记录或订阅详情区域提供低强调的“Request a refund”入口 |

不放置的位置：

- 不放在生成历史列表中。
- 不放在生成结果页。
- 不放在首页主 CTA 附近。
- 用户未主动进入退款申请流程前，不显示“当前可退余额”或“剩余可退款金额”等主动引导退款的字段。

后台必须保留可退款余额字段或可实时计算该值。用户真正发起退款申请后，可以在退款申请/审核流程中向用户展示可退款金额及计算依据。

第二阶段实现时，退款申请可以先采用邮件或简单表单方式：

```text
Request a refund
→ 填写订单号 / 账号邮箱 / 原因
→ 后台人工审核
→ 按积分流水计算可退金额
→ 向用户展示可退款金额和计算依据
→ 通过 Creem.io 原路退款
```

### 7.16 API 成本与毛利记录

生成记录中应保留成本分析字段，便于后续计算毛利：

- 模型。
- 生成类型。
- 消耗积分。
- 估算成本。
- 用户套餐。
- 生成状态。
- 供应商。

第一阶段可以先保留基础字段；第二阶段接入聚合供应商和真实支付后补齐成本与毛利分析。

## 8. 视觉与品牌方向

### 8.1 已确定方向

AiVolo.studio 的视觉方向已经确定：

- 这是娱乐创作产品，不是严肃科技产品。

### 9.8 智能缓存层

第三阶段实现：

- 使用标准化后的 final prompt 做相似度匹配。
- 相似度阈值 85%。
- 缓存结果保存 3 个月。
- 命中缓存后仍显示拟态 loading。
- 用户体验上不区分缓存命中和真实生成。

```text
用户生成请求
  ↓
Prompt 标准化得到 final_prompt
  ↓
计算 embedding
  ↓
查询 generations_cache
  ↓
相似度 >= 85% ?
  ├── 是
  │   ├── 使用缓存结果
  │   ├── 正常扣积分
  │   ├── 显示拟态 loading
  │   └── 返回缓存资产
  │
  └── 否
      ├── 调用真实模型 API
      ├── 保存生成结果
      ├── 判断是否值得缓存
      └── 写入 generations_cache

缓存存储：
├── Supabase Postgres
│   └── generations_cache
│       ├── cache_key
│       ├── template_id
│       ├── final_prompt_hash
│       ├── prompt_embedding
│       ├── aspect
│       ├── result_url
│       ├── cached_at
│       └── expires_at
│
└── 热点缓存层（后续按成本决定是否引入 Redis）
```

## 11. AI 实施模式

本项目正式开发实施阶段由 AI 全流程执行。开发过程采用多 Agent 协作模式，由主执行 Agent 负责整体编排。

实施要求：

- 按产品、设计、前端、后端、数据库、支付、SEO、测试、代码审查等专业方向拆分任务。
- 专业任务交给专业 Agent。
- 能并行的任务并行执行。
- 需要串行的任务严格管理依赖顺序。
- 所有 Agent 产出必须由主执行 Agent 统一整合和验收。
- 代码开发必须遵守专业工程流程和质量门禁。

具体 Agent 分工、执行流程、质量门禁和实时任务状态以 [`DEVELOPMENT_PLAN_V2.md`](./DEVELOPMENT_PLAN_V2.md) 为准。
